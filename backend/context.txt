Communication contract — the golden rule
Synchronous gRPC only when the caller needs a response before it can proceed: Booking → Pricing (need the fare to confirm the ride), Booking → Promo (apply discount before locking price). Everything else — trip lifecycle events, payment triggers, notification dispatch, analytics — goes through Kafka. No service ever queries another service's database.
The Kotlin decision
React Native handles the driver UI fine. But Android background location from JS threads is unreliable — the OS kills it. The fix is a Kotlin foreground service with a persistent notification that keeps GPS alive. RN talks to it via a native module bridge. It's the one place Go can't help you.
Real-time tracking flow
Driver GPS ping → WebSocket Gateway → Tracking service → GEOADD in Redis (sub-millisecond write) → pushed to rider's WebSocket. Kafka gets a durable copy every 5 seconds for replay and analytics. Hot path stays fast; event log stays clean.
Build order for the MVP core loop
Auth → User → Driver → Booking (+ Pricing via gRPC) → Matching → Tracking → Trip → Payment. That's your rideable product. Rating, Promo, Support, and Analytics all consume Kafka events without touching upstream services — add them incrementally with zero upstream changes.
Database per service summary
PostgreSQL for Auth, User, Driver, Booking, Pricing, Promo, Rating, Payment. MongoDB for Trip and Support (flexible document schemas for evolving state machines). Redis for live GPS state, sessions, and rate limits. ClickHouse for Analytics (columnar, fast aggregates).

Why memory-based HPA for the WebSocket gateway
CPU barely moves on a goroutine holding an idle WebSocket connection — but memory climbs linearly with active trips. Scaling on CPU would leave you under-provisioned during peak hours while scaling on memory tracks the actual load precisely.
The PgBouncer requirement is non-negotiable
Go's database/sql opens a connection pool per pod. At 20 Booking pods × 25 connections each = 500 connections hitting PG. PostgreSQL's default max_connections is 100. You'll hit this on day one of a real launch. PgBouncer in transaction mode collapses those 500 app connections down to 20–30 actual PG connections.
Pre-scaling vs reactive HPA
HPA reacts in 15–30 seconds minimum. An IPL match ending at 10:03 PM will dump 50,000 people looking for rides in under 2 minutes. By the time HPA has spun up new Matching pods, you've already lost 30% of those bookings to timeout. The CronJob pattern pre-scales 30 minutes before known events — for unpredictable rain, wire a weather API webhook that patches HPA minReplicas when rainfall exceeds a threshold.
City cluster strategy
Start Hyderabad on a shared Tier-1B regional cluster — you don't need dedicated infra to validate the product. Once you cross ~30k rides/day in a city, migrate it to a dedicated namespace with its own node pool. The geo-partitioned Kafka topics and city_id-keyed Redis from day one mean this migration is a config change, not a re-architecture.
The one thing that will actually kill you in production
Redis going down. Both Matching (GEORADIUS queries) and Tracking (GPS state) are synchronously dependent on it. Run Redis Sentinel with 3 nodes minimum, or Redis Cluster for anything above Tier-1B cities. A single-node Redis is your highest-probability catastrophic failure.

VvisualizeVvisualize show_widgetFive tabs — the process model visualisation on tab 1 shows exactly why Erlang wins here. Here's the critical thinking:
The single biggest win: no sticky sessions
With Go, each WebSocket connection lives in memory on one pod. When pod-2 gets a GPS update from a driver, it has to reach through Redis pub/sub to find the rider who might be on pod-1 or pod-3. That's an extra network hop on every location push. With Phoenix PubSub using the PG2 adapter, all Erlang pods form a native cluster — broadcast! on any pod reaches every subscriber on every pod without touching Redis. Redis is still used for state (GEOADD, driver availability), but the message routing is handled by the BEAM itself.
The CPU vs memory trade-off
Erlang's BEAM scheduler is CPU-hungry compared to Go — give each pod at least 500m CPU request, not 200m. But the memory story is dramatically better: 300 bytes per idle BEAM process vs 8KB minimum goroutine stack in Go. At 100,000 concurrent connections, that's ~30MB vs ~800MB just for the connection bookkeeping. You go from 30 Go pods to 10 Erlang pods for the same connection load.
Use Elixir, not raw Erlang
Phoenix (Elixir framework) gives you Phoenix Channels — a production-grade WebSocket abstraction with presence tracking, reconnect handling, and cluster-aware pub/sub already built. Writing raw Erlang/Cowboy is harder with no meaningful performance gain. Elixir compiles to BEAM bytecode — same runtime, nicer syntax.
The one non-obvious gotcha
The RELEASE_COOKIE must be identical across all Erlang pods in the K8s namespace. It's the shared secret that lets BEAM nodes discover and trust each other. If pods have different cookies, Phoenix PubSub silently fails to broadcast across nodes and you'll spend hours debugging why riders aren't getting location updates from drivers on different pods. Store it in a K8s Secret and inject it as an environment variable at deploy time.
